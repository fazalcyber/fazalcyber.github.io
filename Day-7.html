How Git Works – Explained with Diagram

This documentation explains the core Git workflow. It covers key stages: workspace, staging area, local repository, and remote repository, and how commands like add, commit, push, pull, fetch, and merge interact across these layers.

 Git Workflow Components

1.  Workspace

Your working directory (where your files like .html, .py, .js etc. exist and are edited).

Example files:

.git/

index.html

src/

hello.txt

All coding/editing happens here.

2.  Stage / Staging Area

A temporary holding area where changes are prepared before committing.

· Use git add to stage changes.

· Think of it like selecting files for packing before shipping.

3. Local Repository

A complete Git database (commit history, branches, logs) on your machine.

· Use git commit to record changes to your local repo.

4.  Remote Repository

A shared repository hosted on platforms like:

· �� GitHub

· �� GitLab

This is where your project is stored for collaboration and backup.

 Git Command Flow Breakdown

Command

From → To

Description

git add

Workspace → Staging Area

Stages selected changes

git commit

Staging Area → Local Repository

Saves a snapshot of staged changes

git push

Local Repo → Remote Repository

Uploads your local commits to GitHub/GitLab

git pull

Remote Repo → Workspace

Fetches + merges latest changes

git fetch

Remote Repo → Local Repository

Gets remote changes but doesn’t apply them

git merge

Local Repository → Workspace

Applies fetched changes into your working files

git reset

Removes files from staging area or undo commits

Reverts staged or committed changes

 What git pull Really Does

git pull = git fetch + git merge

· git fetch: Downloads changes from the remote

· git merge: Merges them into your current branch
